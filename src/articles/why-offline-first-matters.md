blocks:
  - type: paragraph
    text: "Useful privacy tools often start by asking a plain question: does this file need to leave the device at all?"
  - type: heading
    text: "Key idea"
  - type: paragraph
    text: "Offline-first software starts with a narrower default: sensitive work stays on the device unless the user deliberately sends it somewhere else."
  - type: heading
    text: "Why the default matters"
  - type: paragraph
    text: "Many modern tools sync files, settings, and activity because that makes cross-device use convenient. Convenience is useful, but it can also create extra copies, account dependencies, server exposure, and recovery paths the user did not actively choose."
  - type: paragraph
    text: "A local-first workflow is easier to reason about. The file begins on the device. Sharing, exporting, or uploading becomes a visible action instead of a quiet background assumption."
  - type: heading
    text: "What this means"
  - type: paragraph
    text: "Fewer services need access to the work by default, and the user has a clearer moment to decide when a file should leave the device."
  - type: heading
    text: "What it does not mean"
  - type: paragraph
    text: "Offline-first is not a guarantee of safety. A compromised phone, weak screen lock, or careless export can still expose private files."
  - type: heading
    text: "Good habits"
  - type: paragraph
    text: "Use local tools when a file does not need collaboration or backup through a cloud account. Keep the phone updated, use a strong device lock, and treat every export as a new copy that needs its own care."
  - type: paragraph
    text: "For private workflows, the benefit is clarity: fewer automatic paths, fewer surprises, and more deliberate decisions."
